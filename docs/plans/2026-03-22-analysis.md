## 2. UI/UX Gaps

### 2.1 Layout & Customization
- **No drag-and-drop widget reordering.** The overview page has a fixed, hardcoded card layout. Modern dashboards (Grafana, Datadog) let users rearrange and resize panels. Even a simple CSS Grid persisted to localStorage would be a significant improvement.
- **No widget visibility toggles.** Users can't hide sections they don't care about (e.g., hide GitHub profile card if not needed).
- **No pinned/favorites dashboard view.** Pinning repos/articles exists, but there's no "pinned items" aggregate view showing all pinned content in one place.

### 2.2 Data Filtering & Exploration
- **No date range picker.** The history endpoint accepts `?days=7` but the UI has no control for this. Users can't explore 30/90-day trends without editing the URL.
- **No global search filters.** The command palette searches current data but has no operator-style filters (e.g., `status:failed`, `stars:>100`).
- **Incidents page has no filter/sort controls.** All incidents are shown in a flat list with no ability to filter by severity, type, or time range.
- **Metrics page has no repo selection.** The bar chart always shows top 8; users can't choose which repos to compare.

### 2.3 Data Tables
- **No pagination.** Repos and articles lists are truncated to a fixed number without infinite scroll or page controls. With >20 repos, content is silently dropped.
- **No column resizing or hiding in tables.** The products table layout is fixed.
- **No bulk actions.** Users can't select multiple agents to bulk-enable/disable.

### 2.4 Empty States & Onboarding
- **No first-run onboarding flow.** A new user hitting the dashboard with no `DEVTO_API_KEY` or `gh` CLI configured gets minimal guidance. Empty states lack actionable CTAs (e.g., "Set your API key in Settings").
- **No onboarding checklist.** No guided setup for new environments.

### 2.5 Notifications
- **No browser push notifications.** Events and incident alerts only surface in-app. There's no way to receive alerts when the tab is closed.
- **No notification preferences.** Users can't configure which events trigger notifications or set severity thresholds.
- **Notification panel has no persistence.** Events shown in the panel come from the backend events log, but the "unread" state resets on page reload.

### 2.6 Real-time UX
- **No visual diff highlighting.** When data refreshes, changed values don't animate or highlight to draw attention. Users can't tell what changed.
- **No "pause auto-refresh" toggle.** While the settings page lets users change the interval, there's no quick toggle to pause polling during manual investigation.
- **SSE reconnection is invisible.** When SSE drops and falls back to polling, there's no user-visible indicator of degraded real-time mode.

### 2.7 Mobile/Responsive Gaps
- **Command palette is not touch-optimized.** Keyboard-centric UX (Cmd+K, g+key sequences) doesn't translate well to mobile.
- **Agent drawer and notification panel may overlap on small screens.** No coordination between simultaneously-open panels.
- **Live logs panel on Agents page is not readable on mobile.** Fixed-height log terminal with horizontal scroll.

## 3. Architecture Issues

### 3.1 Frontend Architecture

**Everything is `"use client"`.** Every page uses the `"use client"` directive, meaning all rendering is client-side. Initial page load shows loading spinners while data fetches complete. Next.js Server Components or Route Handlers with `generateStaticParams` could provide instant initial data, improving perceived performance significantly. At minimum, overview stats could be server-rendered with cached data.

**No centralized state management.** Each page fetches its own data independently. The overview, agents, and metrics pages all fetch agent data separately. If a user navigates between pages, the same API calls repeat. A lightweight store (Zustand, Jotai, or React Context + SWR) would eliminate redundant fetches and share fresh data across pages.

**No API response type definitions.** TypeScript interfaces for API responses are defined inline per-component (or not at all — `any` types appear in some places). A shared `frontend/types/api.ts` would improve type safety and catch backend schema changes at compile time.

**Data fetching logic is duplicated.** The pattern of `useState` + `useEffect` + `fetch` + error handling is repeated in every page component rather than using a consistent data-fetching abstraction. The `useFetchWithTimestamp` hook is good but isn't used consistently everywhere.

**Recharts has no responsive sizing fallback.** The bar chart in metrics uses hardcoded pixel dimensions in some places. Chart containers need `ResponsiveContainer` wrappers everywhere to handle window resizing.

### 3.2 Backend Architecture

**`gh` CLI subprocess calls are fragile.** The GitHub data endpoints call `subprocess.run(["gh", "repo", "list", ...])` and parse stdout. This breaks if `gh` is not on PATH, if authentication expires, or if output format changes. The official GitHub REST API (via `httpx`) would be more reliable and testable.

**TTL cache is in-memory only.** The `TTLCache` class is thread-safe in-memory but is wiped on every restart. During development (frequent restarts), this means constant API hammering against GitHub and dev.to rate limits. A disk-backed cache (even just a JSON file) or Redis would survive restarts.

**Synchronous incident aggregation.** The `/api/incidents` endpoint calls multiple other internal functions synchronously and aggregates results inline. With many agents, this could be slow. A background task that periodically computes and caches incident state would be faster.

**Events log is append-only with no rotation.** The events are written to a JSONL file that grows unbounded. Heavy cron activity could create very large event logs over time. The `/api/events/clear` endpoint is the only cleanup mechanism, and it requires manual invocation.

**SQLite history has no schema migration.** The `history.db` table is created with `CREATE TABLE IF NOT EXISTS` but there's no migration system. Adding columns requires dropping and recreating the table.

**No request validation on action endpoints.** `POST /api/crons/{id}/run` doesn't validate that the ID exists before attempting to run. It returns 500 on unknown IDs rather than 404.

**No structured backend logging.** `print()` statements are used for logging. A proper logger with levels, timestamps, and structured output (JSON) would make debugging easier and enable log aggregation.

**Missing async parallelism in aggregation endpoints.** The `/api/overview` endpoint fetches repos, articles, and agents sequentially. Using `asyncio.gather()` would parallelize these and reduce response time by ~2–3x.

### 3.3 Security

**`DASHBOARD_API_KEY` is optional and has no enforcement UI.** The backend can run completely open. There's no warning in the health bar or settings page indicating that authentication is disabled.

**README endpoint exposes arbitrary repo file content.** `/api/products/{repo}/readme` fetches any repo's README. While scoped to the authenticated user's repos, there's no content sanitization on the returned markdown.

**SSE endpoint has no authentication bypass protection.** The SSE endpoint (`/api/sse`) is excluded from auth middleware checks, allowing unauthenticated access to real-time event streams when `DASHBOARD_API_KEY` is set.

## 4. Missing Operational Features

### 4.1 Alerting & Notifications
- **No configurable alerting rules.** Users can't define "notify me when any agent fails 3 times in a row" or "alert when stars drop" or "warn when article views < X." The incidents page is reactive, not proactive.
- **No alert silencing/suppression.** Known failures (e.g., a deliberately paused cron) generate noise in the incidents list with no way to silence them.
- **No alert history / acknowledgment workflow.** There's no record of which alerts were seen and acted upon.

### 4.2 SLA & Reliability Tracking
- **No uptime tracking per agent.** The agents page shows current status but no historical uptime percentage (e.g., "99.2% success rate over 30 days"). This is critical for operational dashboards.
- **No MTTR/MTTF metrics.** Mean Time to Recovery and Mean Time to Failure aren't computed or displayed.
- **No SLO definition UI.** Users can't set targets like "this agent should run successfully 95% of the time" and track drift toward or away from that target.

### 4.3 Agent Orchestration Visualization
- **No dependency graph.** If agents have dependencies (e.g., agent B only runs after agent A), there's no visualization of this. A DAG view showing agent relationships would be valuable.
- **No execution timeline / Gantt view.** There's no chronological view of when agents ran, for how long, and whether they overlapped.
- **No agent grouping / tagging.** Agents can't be grouped by type, environment, or team. With many agents, the flat table becomes unwieldy.
- **No agent run comparison.** Users can't compare two runs of the same agent to see what changed (duration, output diff, etc.).

### 4.4 Trend Analysis
- **No week-over-week / month-over-month comparison.** The history endpoint provides 7-day snapshots but the UI doesn't show comparative trends. "Stars this week vs last week" would immediately reveal growth or decline.
- **No anomaly detection.** Unusually slow agent runs or sudden drops in article views aren't flagged automatically.
- **No forecasting.** Even simple linear projections ("at current growth, you'll hit 1k stars in X days") would add operational value.

### 4.5 Audit & Traceability
- **No audit log for manual actions.** When a user triggers a cron run, enables/disables an agent, or clears events, there's no record of who did it and when. For operational dashboards, auditability is essential.
- **No run annotation system.** Users can't add notes to incidents or agent runs (e.g., "manually retried after fixing config").

### 4.6 External Integrations
- **No webhook delivery.** There's no way to push events/alerts to Slack, Discord, PagerDuty, or email. The notification system is entirely in-app.
- **No CI/CD integration beyond status.** CI status badges show pass/fail but there's no link to logs, no ability to trigger reruns, and no deployment history tracking.
- **No cost tracking.** No visibility into GitHub Actions minutes consumed, API rate limit usage, or any cost-related metrics.

## 5. Data Visualization Gaps

### 5.1 Current Charts
The dashboard currently has exactly **one chart**: a Recharts bar chart on the metrics page showing top 8 repos by stars vs. forks. The sparklines in stat cards are the only other visual data encoding.

### 5.2 High-Value Missing Visualizations

**Time-series / Line Charts:**
- Stars over time per repo (using history data already collected in SQLite)
- Article views trajectory (daily/weekly rollups)
- Agent success rate trend over time
- Notification/event volume over time

**Activity Heatmap:**
- GitHub contribution-style heatmap showing agent run frequency and success rate across days/hours
- Article publication cadence heatmap
- This is especially valuable for identifying patterns (e.g., agents always fail on Mondays)

**Distribution Charts:**
- Donut chart: breakdown of agent statuses (active/paused/failing)
- Donut chart: article views distribution across publications
- Histogram: agent run duration distribution (p50/p90/p99)

**Agent Timeline / Gantt:**
- Horizontal bar chart showing recent agent executions by time, colored by success/failure
- Shows overlap, gaps, and patterns in execution scheduling

**Scatter/Bubble Charts:**
- Repos plotted by stars vs. forks vs. age (bubble = recency)
- Articles by views vs. reactions vs. age

**Stat Comparisons:**
- Period-over-period comparison cards: "stars +12 this week (+3.4%)"
- Before/after incident correlation

### 5.3 Data Already Available for New Visualizations
The SQLite `history.db` already stores snapshot data. The backend can already compute: stars/forks counts, article views/reactions, agent run counts. The building blocks exist — the frontend just doesn't use them beyond the 4 sparklines.

## 6. Developer Experience Gaps

### 6.1 Testing

**No test files exist in the repository.** Despite `vitest` and `playwright` being listed in `package.json` devDependencies, there are no test files discoverable in the codebase. The `requirements-dev.txt` includes `pytest` and `pytest-asyncio` but there are no Python test files either.

**No test infrastructure configured:**
- No `vitest.config.ts`
- No `playwright.config.ts`
- No test fixtures or factories
- No mock server for API testing

**Critical untested paths:**
- SSE reconnection behavior
- Cache TTL expiration
- Incident detection logic
- Cron enable/disable/run API actions
- localStorage settings persistence

### 6.2 CI/CD

**No CI/CD pipeline.** There are no `.github/workflows/` files. For a project tracking other repos' CI status, it's notable that this repo has none itself. Missing:
- Lint check (ESLint/ruff)
- Type check (`tsc --noEmit`)
- Unit tests
- E2E smoke test
- Docker build verification

### 6.3 Documentation

**No README at project root.** There's no top-level README explaining how to set up and run the project. A new contributor must infer the setup from `package.json` and `requirements.txt`.

**No API documentation.** FastAPI auto-generates OpenAPI docs at `/docs`, but it's unclear if this is enabled in the deployed configuration. The endpoint catalog (30+ endpoints) has no external documentation.

**No CLAUDE.md at project root.** (There may be one — not confirmed.) Without it, AI coding assistants lack project-specific conventions.

**Inline comments are sparse.** The backend `main.py` has some DASH-0XX ticket references in comments but business logic is largely uncommented. The cron log parsing and incident detection algorithms would benefit from documentation.

### 6.4 Developer Tooling

**No `docker-compose.yml`.** Running the full stack locally requires manually starting the backend (`uvicorn`) and frontend (`next dev`) in separate terminals. A compose file would enable one-command startup.

**No `Makefile` or task runner.** Common tasks (start backend, start frontend, run tests, reset DB) have no automation.

**No pre-commit hooks.** Linting and type-checking aren't enforced at commit time. The modified files in the current git status (`shortcuts-help-modal.tsx`, `sidebar.tsx`, `shortcuts.ts`) would benefit from automated checks.

**No environment variable validation.** The backend starts successfully even if `DEVTO_API_KEY` is missing, silently returning empty article data. A startup validation check with clear error messages would improve DX.

**No hot-reload for backend.** `uvicorn` with `--reload` is available but not the default invocation. Backend changes require manual restarts.

## 7. Priority Recommendations

### High Priority (Low Effort, High Impact)
1. **Add `asyncio.gather()` in `/api/overview`** — ~5 min change, 2–3x response time improvement
2. **Create shared `frontend/types/api.ts`** — eliminate `any` types, catch schema drift
3. **Add date range picker to history views** — the data exists, just needs UI controls
4. **Add period-over-period comparison to stat cards** — "stars +12 this week"
5. **Configure `pytest` with basic backend unit tests** — at least test incident detection and cache logic

### Medium Priority (Moderate Effort, High Value)
6. **Replace `gh` CLI subprocess with GitHub REST API via `httpx`** — more reliable, testable, no PATH dependency
7. **Add agent execution timeline chart** — Recharts `BarChart` with custom time axis
8. **Add configurable alerting rules** — simple JSON config defining threshold-based alerts
9. **Add GitHub Actions CI workflow** — lint + typecheck + tests on push
10. **Add activity heatmap** — cal-heatmap or custom SVG grid using existing history data

### Lower Priority (Higher Effort, Architectural)
11. **Introduce SWR or React Query** — replace per-page `useState/useEffect` fetching with shared cache
12. **Add SQLite schema migrations** — Alembic or simple version table
13. **Implement alert silencing/acknowledgment** — backend events flag + frontend UI
14. **Add drag-and-drop overview layout** — `@dnd-kit/core` with localStorage persistence
15. **Add dependency graph visualization** — D3 or React Flow for agent DAG
